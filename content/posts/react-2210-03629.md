> 论文：**ReAct: Synergizing Reasoning and Acting in Language Models**（arXiv:2210.03629）

如果你只想记一句话：
**ReAct = 推理（Reasoning）+ 行动（Acting）+ 观察反馈（Observation）**。

![ReAct 封面图](images/react-2210-03629/01-封面-React是什么.png)

---

## 为什么会有 ReAct？

在 ReAct 之前，主流有两条路：

1. **只强调推理**（如 CoT）
   - 优点：思路看起来更完整
   - 问题：容易“在错误前提上越想越远”

2. **只强调行动**（直接调用工具/环境操作）
   - 优点：执行快
   - 问题：容易盲动，缺少全局判断

ReAct 的关键创新，就是把这两条路接成闭环：

**思考 → 行动 → 观察 → 修正思考**

不是一次性想完再做，而是**边做边改**。

![ReAct 核心循环](images/react-2210-03629/02-核心循环-ThoughtActionObservation.png)

---

## ReAct 的核心机制（人话版）

在每一步里，模型会做三件事：

- **Thought（思考）**：现在我该怎么走下一步？
- **Action（行动）**：去查资料 / 调工具 / 执行操作
- **Observation（观察）**：拿到外部反馈，再决定是否修正

这意味着模型不再只靠“脑补”，而是持续用外部世界校准自己。

---

## 和 CoT 的本质区别

CoT 更像：
> 我先在脑子里一路推到答案

ReAct 更像：
> 我先想一步，做一步，看结果，再修正

所以 ReAct 在真实任务中通常更抗偏航，因为它有“回看”和“纠错”。

![CoT 与 ReAct 对比](images/react-2210-03629/03-对比-CoTvsReAct.png)

---

## 论文里它强在哪里？

论文在多类任务上验证了 ReAct 的价值：

- **多跳问答**：需要多步检索与串联证据
- **事实核验**：减少凭空编造
- **交互决策任务**：需要动作-反馈-再动作
- **网页/环境操作类任务**：更像真实 agent 场景

总体结论很清晰：
**ReAct 不只是“答得像”，而是“做得稳”。**

![论文效果场景](images/react-2210-03629/04-论文结果-任务表现.png)

## 给 Agent 工程落地的 4 条建议

### 1) 把“工具调用”嵌进推理循环
不要把“思考模块”和“工具模块”割裂开。

### 2) 每次行动后强制观察校验
下一步必须基于 Observation，而不是复读旧计划。

### 3) 必须设置停止条件
至少包含：
- 最大步数
- 置信度阈值
- 连续失败上限

### 4) 保留中间轨迹用于复盘
留 Thought/Action/Observation 日志，调参效率会高很多。

![Agent 落地模板](images/react-2210-03629/05-工程落地-Agent模板.png)

---

## ReAct 的风险与护栏

常见风险：
- 一直循环调用同类工具
- 在低价值动作里空转
- 行动结果没被正确利用

对应护栏：
- 工具白名单 + 参数约束
- 每步价值评估（这一步是否让答案更接近目标）
- 达到阈值立刻停止并总结

一句话：
**会推理很重要，会刹车更重要。**

![风险与护栏](images/react-2210-03629/06-风险与护栏-防死循环.png)

---

## 一句话总结

ReAct 的意义，不是让模型更会“说”，而是让模型更会“做”。

它把 LLM 从“静态回答器”推进到“动态问题解决者”。
如果你在做 agent，这篇论文几乎是必读。
