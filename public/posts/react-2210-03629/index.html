论文：ReAct: Synergizing Reasoning and Acting in Language Models（arXiv:2210.03629）

先把结论摆前面： ReAct 的价值不在“多一个提示词技巧”，而在把 LLM 从“静态回答”升级成“闭环决策系统”。

0. 为什么我说“只会 CoT 已经不够了”#

在真实任务里（检索、工具调用、网页交互、流程决策），你会遇到三类典型失败：

  1. 推理正确，但事实错（幻觉）
  2. 计划看着很完整，但执行后不回看（错误传播）
  3. 会调用工具，但不知道何时该停、何时该改方向

CoT 主要解决第 1 层“展开推理”的问题； ReAct 解决的是第 2、3 层“推理-执行-反馈”闭环问题。

1. 论文到底提出了什么（不是口号版）#

论文做的事可以压成一句工程话：

把 LLM 的输出拆成 Thought（内部推理）和 Action（可执行动作），并把外部环境返回的 Observation 喂回下一轮推理。

形成循环：

Thought → Action → Observation → Thought → … → Final Answer

这比“让模型多想想”多了两件关键能力：

  • 对外部世界取证（不只靠参数记忆）
  • 基于反馈在线修正策略（不是一条路走到黑）

2. ReAct 为什么比 CoT 更像“可用系统”#

CoT 的强项#

  • 内部推理展开，适合算术/符号推理等“封闭题目”

CoT 的弱项#

  • 外部事实不可验证
  • 一旦前提错，后面会很“自洽地错下去”

ReAct 的强项#

  • 推理过程中允许查证
  • 执行后可回看并修正
  • 轨迹可审计（这点对 Agent 运维特别重要）

3. 论文实验：它在哪些任务上证明了价值#

论文不是只测一种题，而是覆盖两大类：

A. 语言任务（如 HotpotQA、FEVER）#

目标：多跳问答、事实核验

  • 纯 CoT 容易“编合理故事”
  • ReAct 通过外部检索（如 Wikipedia API）把关键链路落到证据上

收益：

  • 幻觉减少
  • 错误传播降低
  • 推理过程更可解释

B. 交互决策任务（如 ALFWorld、WebShop）#

目标：在环境中一步步行动到达目标

  • 只做 action 的 agent 易盲动
  • ReAct 在每步后引入思考与反省，策略更稳定

收益：

  • 长链路任务成功率更高
  • 异常场景恢复能力更好

4. 你可以直接抄的 ReAct 模板（工程可用）#

下面这版是“能上线”的，不是课堂 demo。

你是一个会使用工具的问题解决代理。
每一轮严格输出以下之一：
1) Thought: 你的当前判断（简洁）
2) Action: 工具名[参数]
3) Final Answer: 最终答案

规则：
- 先确认目标，再决定是否需要工具。
- 每次 Action 后，必须基于 Observation 更新 Thought。
- 若证据不足，不得直接 Final Answer。
- 达到停止条件（最大步数/证据充分/连续失败）时结束。

我建议再加三条护栏：

  • max_steps（如 8~12）
  • max_same_tool_repeat（同工具重复上限）
  • evidence_threshold（证据充分度阈值）

5. ReAct 的失败模式（很多文章不讲这个）#

失败模式 1：循环空转#

症状：不停 Thought/Action，但信息增益接近 0

解法：

  • 引入“步进价值评估”：本步是否实质减少不确定性
  • 连续低价值动作直接 early stop

失败模式 2：动作正确，观察没吸收#

症状：工具返回关键证据，但下一步推理没用上

解法：

  • 强制 Observation 摘要结构化（fact / confidence / source）
  • 下一步 Thought 必须引用 Observation 中字段

失败模式 3：过度工具依赖#

症状：明明可直接回答，仍频繁查工具

解法：

  • 加“工具调用成本”惩罚
  • 明确“何时不该调用工具”

6. 和现在主流 Agent 框架的关系#

你会发现很多现代 Agent（函数调用、多工具编排、Planner-Executor）本质都在实现 ReAct 思想：

  • 规划不是一次性离线完成
  • 执行不是盲目流水线
  • 每步都要吃反馈并更新策略

换句话说： ReAct 更像“范式底座”，而不是单一 prompt 技巧。

7. 给实战团队的落地清单（可直接贴 Jira）#

  1. 定义 Thought / Action / Observation 的统一日志 schema
  2. 给每类工具定义可观测返回结构（别只返回 raw text）
  3. 设置 stop policy（步数、失败、置信度）
  4. 加 replay & trace 页面（用于复盘）
  5. 先在高价值链路试点（检索问答、工单诊断、流程决策）

8. 最后一句#

如果 CoT 让模型“更会想”， 那 ReAct 让模型“想得对、做得到、改得动”。

这就是它在 2022 年提出后，为什么到今天还在 Agent 工程里有长期价值。
